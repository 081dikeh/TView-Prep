HTML & CSS Questions
Scalability, architecture, cross-browser
Q01 What best practices should be followed for scalable and maintainable HTML/CSS? +

Use semantic HTML elements (e.g. <article>, <section>, <nav>) so structure is meaningful to both browsers and screen readers.

Adopt a modular CSS architecture like BEM (Block Element Modifier) or SMACSS to keep class names predictable and collision-free.

Use pre-processors like SASS/SCSS to organise CSS with variables, nesting, and mixins. Keep specificity low — avoid !important and overly nested selectors.

Enforce naming conventions, organise files logically (by feature/component), document reusable patterns, and use version control + code reviews to maintain standards.

BEM example: .card__title--highlighted — Block: card, Element: title, Modifier: highlighted.
Q02 How do you ensure cross-browser compatibility in complex front-end projects? +

Write standard-compliant HTML5 and CSS3. Use feature detection (not browser detection) via libraries like Modernizr.

Use CSS vendor prefixes for experimental properties, and tools like Autoprefixer to automate this. Use normalize.css to reset inconsistent browser defaults.

Add polyfills for unsupported features (e.g., fetch, IntersectionObserver), and test rigorously on Chrome, Firefox, Safari, and Edge using real devices or BrowserStack.

Q03 How do you structure CSS for large-scale responsive web applications? +

Use BEM for naming, a component-driven architecture (one file per component), and SASS/SCSS for variables, mixins, and nested organisation.

Follow a mobile-first approach: write base styles for small screens, then use min-width media queries to scale up. This ensures progressively enhanced experiences.

Keep media queries modularised inside their component files rather than in a single giant file — this keeps styles co-located and maintainable.

Mobile-first media query: @media (min-width: 768px) { ... } — base styles are for small screens.
Q04 What are the principles of progressive enhancement vs graceful degradation? +

Progressive Enhancement: Start with a baseline that works everywhere — plain HTML with basic CSS. Then layer in advanced features (CSS animations, JS interactivity) for browsers that support them. Everyone gets a functional experience.

Graceful Degradation: Build for modern browsers first, then ensure it still functions in older/less-capable ones — even if some features are missing or simplified.

In practice, progressive enhancement is preferred because it's accessibility-first and more inclusive. Use feature queries (@supports) to conditionally apply advanced CSS.

Q05 Explain the CSS Box Model and how box-sizing affects layout. +

Every element is a box with: content → padding → border → margin.

With box-sizing: content-box (default), width/height refers to content only — padding and border are added on top, making the actual rendered size larger than declared.

With box-sizing: border-box, width/height includes padding and border. This is almost always what you want and is why most stylesheets set:

/* Apply border-box globally */
*, *::before, *::after {
  box-sizing: border-box;
}
Q06 When would you use CSS Grid vs Flexbox? +

Flexbox is for one-dimensional layout — aligning items in a row or column. Great for navbars, button groups, centering, and card rows.

CSS Grid is for two-dimensional layout — rows AND columns simultaneously. Great for page layouts, dashboards, and complex card grids.

Rule of thumb: If you're thinking about rows OR columns → Flexbox. If you're thinking about rows AND columns → Grid. They're often used together.
/* Flexbox: horizontal nav */
.nav { display: flex; gap: 16px; align-items: center; }

/* Grid: 3-col card layout */
.grid { display: grid; grid-template-columns: repeat(3, 1fr); gap: 24px; }

JavaScript Questions
Core concepts, async, closures, ES6+
Q07 What advanced JavaScript concepts are essential for front-end development? +
  • Closures — functions that remember the scope they were created in
  • Asynchronous programming — Promises, async/await, and the event loop
  • Prototypal inheritance — how objects inherit via the prototype chain
  • Event delegation — attaching one listener to a parent instead of many children
  • ES6+ features — destructuring, spread/rest, optional chaining, nullish coalescing
  • Functional programming — pure functions, immutability, map/filter/reduce
  • Module patterns — ES modules, import/export, code splitting
Q08 Explain closures in JavaScript with an example. +

A closure is a function that has access to its outer (enclosing) function's variables even after that outer function has returned.

function makeCounter() {
  let count = 0; // private variable
  return {
    increment: () => ++count,
    decrement: () => --count,
    value: () => count
  };
}

const counter = makeCounter();
counter.increment(); // 1
counter.increment(); // 2
counter.value();    // 2

Practical uses: module pattern, data encapsulation, memoisation, event handlers with captured state.

Q09 How does the JavaScript event loop work? +

JavaScript is single-threaded. The event loop allows async operations by managing three things:

  • Call Stack — executes synchronous code (LIFO)
  • Web APIs / Task Queue — setTimeout, DOM events, fetch callbacks wait here
  • Microtask Queue — Promise callbacks (.then, .catch) — processed before the task queue

Order: synchronous code → microtasks (Promises) → macrotasks (setTimeout, setInterval).

console.log('1');                         // sync
setTimeout(() => console.log('2'), 0);  // macrotask
Promise.resolve().then(() => console.log('3')); // microtask
console.log('4');                         // sync
// Output: 1, 4, 3, 2
Q10 Explain async/await vs Promises. When would you use each? +

Promises represent an eventual value. They use .then()/.catch() chaining. Good for multiple parallel operations via Promise.all().

async/await is syntactic sugar over Promises — it makes async code read like synchronous code. Easier to reason about for sequential steps.

// Promises
fetchUser(id)
  .then(user => fetchPosts(user.id))
  .then(posts => render(posts))
  .catch(err => console.error(err));

// async/await (cleaner)
async function loadUserPosts(id) {
  try {
    const user = await fetchUser(id);
    const posts = await fetchPosts(user.id);
    render(posts);
  } catch (err) {
    console.error(err);
  }
}
Parallelism: Use Promise.all([a(), b()]) to run tasks concurrently instead of sequentially.
Q11 What is event delegation and why is it useful? +

Instead of attaching event listeners to every individual child element, you attach one listener to a common parent and use event.target to identify which child was clicked. This leverages event bubbling.

// Without delegation — N listeners
document.querySelectorAll('.btn').forEach(btn =>
  btn.addEventListener('click', handleClick)
);

// With delegation — 1 listener
document.querySelector('#list').addEventListener('click', (e) => {
  if (e.target.matches('.btn')) handleClick(e);
});

Benefits: Better performance with large lists, works on dynamically added elements, lower memory usage.

React & State Management
Hooks, Context, Redux, data flow
Q12 How do you manage state and handle data flow in large React applications? +

State management scales from local to global depending on complexity:

  • useState / useReducer — local component state
  • Context API — share state across a subtree without prop drilling
  • Redux / Zustand / Jotai — centralised global state for large apps

Key principles: immutability (never mutate state directly), single source of truth, predictable state transitions, and middleware (like Redux Thunk or Saga) for side effects like API calls.

Rule: Don't use Redux for everything. Start with useState → Context → Redux only when Context becomes a performance bottleneck or logic is complex.
Q13 Explain React's key hooks: useState, useEffect, useMemo, useCallback. +

useState — declares reactive state. Re-renders component on update.

useEffect — runs side effects (data fetching, subscriptions) after render. The dependency array controls when it runs.

useMemo — memoises a computed value. Only recomputes when dependencies change. Avoids expensive recalculations on every render.

useCallback — memoises a function reference. Useful when passing callbacks to child components to prevent unnecessary re-renders.

// useMemo example
const sortedList = useMemo(() =>
  [...items].sort((a, b) => a.name.localeCompare(b.name)),
  [items] // only recomputes when items changes
);

// useCallback example
const handleClick = useCallback(() => {
  doSomething(id);
}, [id]);
Q14 What is React's reconciliation and how does the Virtual DOM work? +

React maintains a Virtual DOM — a lightweight JS representation of the actual DOM. When state or props change, React creates a new Virtual DOM tree.

The reconciliation algorithm (diffing) compares the old and new Virtual DOM trees and calculates the minimum set of changes needed. It then batches those changes and applies them to the real DOM efficiently.

The key prop is critical in lists — it tells React which items changed, were added, or removed, enabling accurate diffing without re-rendering everything.

Q15 What is the difference between controlled and uncontrolled components? +

Controlled component: Form data is managed by React state. The component's value is always in sync with state. This is the React-recommended approach.

Uncontrolled component: Form data is managed by the DOM itself. You access it via a ref rather than state. Useful for file inputs or integrating with non-React libraries.

// Controlled
const [value, setValue] = useState('');
<input value={value} onChange={e => setValue(e.target.value)} />

// Uncontrolled
const inputRef = useRef(null);
<input ref={inputRef} />
// Access value: inputRef.current.value

Performance Optimisation
Images, bundles, rendering, caching
Q16 What are your strategies for front-end web performance optimisation? +
  • Minify HTML, CSS, JS (Webpack, Vite do this automatically in production)
  • Lazy loading — use loading="lazy" on images, dynamic import() for JS modules
  • Code splitting — split bundles by route or feature so users only download what they need
  • Image optimisation — use WebP/AVIF format, serve correct sizes with srcset, compress with tools like Squoosh
  • CDN — serve static assets from edge locations geographically close to users
  • Browser caching — set appropriate Cache-Control headers, use content hashing in filenames
  • Reduce HTTP requests — bundle files, use CSS sprites, inline critical CSS
  • Defer non-critical JS — use <script defer> or <script async>
Measure first! Use Lighthouse, Chrome DevTools Performance tab, and Web Vitals (LCP, FID, CLS) to identify real bottlenecks before optimising.
Q17 How would you optimise images on a web page? +
  • Format: Use WebP or AVIF (smaller than PNG/JPEG). Use SVG for icons and illustrations.
  • Size: Serve appropriately sized images using the srcset attribute and sizes descriptor.
  • Lazy loading: Add loading="lazy" to images below the fold.
  • Compression: Compress without visible quality loss using tools like Squoosh, ImageOptim, or Sharp (Node.js).
  • CDN: Use image CDNs like Cloudinary or Imgix that auto-optimise and resize on demand.
<img
  src="hero-800.webp"
  srcset="hero-400.webp 400w, hero-800.webp 800w, hero-1200.webp 1200w"
  sizes="(max-width: 600px) 400px, 800px"
  loading="lazy"
  alt="Hero image"
/>

Q19 What techniques are effective for debugging complex front-end issues? +
  • Chrome DevTools — Elements tab for DOM, Console for JS errors, Network tab for API calls, Performance tab for bottlenecks
  • Breakpoints — pause execution at any line in the Sources tab
  • React DevTools — inspect component tree, props, state, and re-renders

  • console.log strategically — but clean up before committing!
Q20 What are the strategies for modularising JavaScript in enterprise projects? +

Use ES6 modules (import/export) with bundlers like Webpack or Vite. Each module has one responsibility and minimal dependencies.

Organise by feature/domain rather than type (avoid: /components, /utils — prefer: /auth, /dashboard, each containing its own component, hook, and util files).

Use barrel files (index.js) to expose clean public APIs from each folder. Document reusable modules with JSDoc or TypeScript interfaces.

Architecture & System Design
SSR, CSR, Next.js, micro-frontends
Q21 What are the trade-offs between SSR, CSR, and SSG? +

CSR (Client-Side Rendering) — HTML is rendered in the browser by JavaScript. Fast subsequent navigation but slow initial load and poor SEO without extra work. Best for dashboards and apps behind auth.

SSR (Server-Side Rendering) — HTML is generated on the server per request. Fast initial paint, great SEO, but higher server load. Best for pages with dynamic, user-specific content.

SSG (Static Site Generation) — HTML is pre-generated at build time. Blazing fast, cached at CDN edge, great SEO. Best for blogs, marketing pages, documentation.

Next.js lets you mix all three — you can SSG most pages and use SSR or ISR (Incremental Static Regeneration) for pages that need fresh data.
Q22 How would you solve a specific structural problem in a large front-end codebase? +

This is a common open-ended question. Structure your answer using a STAR-like approach:

  1. Identify the problem — "The codebase had no separation between API logic and UI components, leading to tight coupling and untestability."
  2. Propose the solution — "I introduced a service layer pattern — all API calls moved to dedicated service files, components consumed hooks that wrapped those services."
  3. Show the impact — "This made components testable with mocked services, and reduced duplicate API calls by 40%."
Tip: Always mention a real example from your experience. Generic answers score lower than specific, confident ones.

EASY 1. Debounce Function
Implement a debounce(fn, delay) function that delays invoking fn until after delay ms have elapsed since the last call.
// Implement this function:
function debounce(fn, delay) {
  // your code here
}

// Example usage:
const search = debounce((query) => {
  console.log('Searching:', query);
}, 300);

search('h');
search('he');
search('hel');
// Only 'hel' should trigger after 300ms
function debounce(fn, delay) {
  let timer;
  return function(...args) {
    clearTimeout(timer);             // cancel previous timer
    timer = setTimeout(() => {
      fn.apply(this, args);           // call with correct context
    }, delay);
  };
}

// Key concepts: closures, setTimeout, clearTimeout
// Real use case: search inputs, resize handlers, scroll events
Talk through it: "I'm using a closure to persist the timer ID between calls. Each new call cancels the previous timer with clearTimeout, so the function only fires after the caller has stopped for 'delay' ms."
EASY 2. Flatten a Nested Array
Write a function to flatten a deeply nested array without using Array.flat().
flatten([1, [2, [3, [4]], 5]]);
// Expected: [1, 2, 3, 4, 5]
function flatten(arr) {
  return arr.reduce((flat, item) =>
    flat.concat(
      Array.isArray(item) ? flatten(item) : item
    ),
  []);
}

// Alternative using recursion + spread:
function flatten2(arr) {
  return arr.reduce((acc, val) =>
    Array.isArray(val)
      ? [...acc, ...flatten2(val)]
      : [...acc, val],
  []);
}

MEDIUM 4. Custom useFetch Hook
Build a reusable useFetch(url) React hook that handles loading, data, and error states.
// Build a hook that can be used like this:
const { data, loading, error } = useFetch('https://api.example.com/users');

// Should:
// - Start loading when url changes
// - Handle errors gracefully
// - Clean up on unmount
import { useState, useEffect } from 'react';

function useFetch(url) {
  const [data, setData] = useState(null);
  const [loading, setLoading] = useState(true);
  const [error, setError] = useState(null);

  useEffect(() => {
    let cancelled = false; // prevent state update on unmount

    const fetchData = async () => {
      setLoading(true);
      setError(null);
      try {
        const res = await fetch(url);
        if (!res.ok) throw new Error(`HTTP error: ${res.status}`);
        const json = await res.json();
        if (!cancelled) setData(json);
      } catch (err) {
        if (!cancelled) setError(err.message);
      } finally {
        if (!cancelled) setLoading(false);
      }
    };

    fetchData();
    return () => { cancelled = true; }; // cleanup
  }, [url]);

  return { data, loading, error };
}
Key talking point: The cancelled flag prevents state updates after the component unmounts, which would cause a React memory leak warning.
MEDIUM 5. Throttle Function
Implement throttle(fn, limit) — ensures fn is called at most once every limit milliseconds regardless of how many times it's triggered.
// Throttle vs Debounce:
// Debounce: wait N ms after LAST call
// Throttle: fire at most once per N ms
function throttle(fn, limit) {
  // your code here
}
function throttle(fn, limit) {
  let lastCall = 0;
  return function(...args) {
    const now = Date.now();
    if (now - lastCall >= limit) {
      lastCall = now;
      return fn.apply(this, args);
    }
  };
}

// Use case: scroll handlers, window resize, button spam protection
window.addEventListener('scroll', throttle(handleScroll, 100));
MEDIUM 6. Todo List (React)
Build a functional React Todo list with add, toggle complete, and delete functionality. This is the classic micro1 live coding task.
// Requirements:
// 1. Input field + "Add" button
// 2. List of todos, each with:
//    - Text
//    - Checkbox to toggle done
//    - Delete button
// 3. Completed todos styled differently
import { useState } from 'react';

export default function TodoApp() {
  const [todos, setTodos] = useState([]);
  const [input, setInput] = useState('');

  const addTodo = () => {
    if (!input.trim()) return;
    setTodos(prev => [
      ...prev,
      { id: Date.now(), text: input.trim(), done: false }
    ]);
    setInput('');
  };

  const toggleTodo = (id) =>
    setTodos(prev => prev.map(t =>
      t.id === id ? { ...t, done: !t.done } : t
    ));

  const deleteTodo = (id) =>
    setTodos(prev => prev.filter(t => t.id !== id));

  return (
    <div>
      <input
        value={input}
        onChange={e => setInput(e.target.value)}
        onKeyDown={e => e.key === 'Enter' && addTodo()}
        placeholder="Add a task..."
      />
      <button onClick={addTodo}>Add</button>
      <ul>
        {todos.map(todo => (
          <li key={todo.id}
            style={{ textDecoration: todo.done ? 'line-through' : 'none' }}>
            <input type="checkbox" checked={todo.done}
              onChange={() => toggleTodo(todo.id)} />
            {todo.text}
            <button onClick={() => deleteTodo(todo.id)}>✕</button>
          </li>
        ))}
      </ul>
    </div>
  );
}

HARD 7. Virtualised List Component
Implement a basic virtualised list that only renders visible items — for a list of 10,000 items where each row is 40px tall.
// The problem: rendering 10,000 DOM nodes = browser crash
// The solution: only render what's visible in the viewport
// 
// Given:
// - containerHeight = 400px
// - rowHeight = 40px
// - items = array of 10,000 items
// 
// Only ~10 rows are visible at a time
import { useState, useRef } from 'react';

function VirtualList({ items, rowHeight = 40, containerHeight = 400 }) {
  const [scrollTop, setScrollTop] = useState(0);

  const totalHeight = items.length * rowHeight;
  const startIdx = Math.floor(scrollTop / rowHeight);
  const visibleCount = Math.ceil(containerHeight / rowHeight) + 1; // +1 buffer
  const endIdx = Math.min(startIdx + visibleCount, items.length);
  const visibleItems = items.slice(startIdx, endIdx);

  return (
    <div
      style={{ height: containerHeight, overflowY: 'auto', position: 'relative' }}
      onScroll={e => setScrollTop(e.target.scrollTop)}
    >
      {/* Spacer to maintain scroll height */}
      <div style={{ height: totalHeight, position: 'relative' }}>
        {visibleItems.map((item, i) => (
          <div
            key={startIdx + i}
            style={{
              position: 'absolute',
              top: (startIdx + i) * rowHeight,
              height: rowHeight,
              width: '100%',
            }}
          >
            {item}
          </div>
        ))}
      </div>
    </div>
  );
}
Note: In production you'd use react-window or react-virtual. But knowing the concept demonstrates deep performance understanding.
HARD 8. Implement Promise.all from scratch
Implement your own version of Promise.all() that resolves when all promises resolve, or rejects if any promise rejects.
myPromiseAll([
  Promise.resolve(1),
  Promise.resolve(2),
  Promise.resolve(3),
]).then(console.log); // [1, 2, 3]
function myPromiseAll(promises) {
  return new Promise((resolve, reject) => {
    if (!promises.length) return resolve([]);

    const results = new Array(promises.length);
    let completed = 0;

    promises.forEach((promise, i) => {
      Promise.resolve(promise)  // handle non-promise values
        .then(value => {
          results[i] = value;      // preserve order
          completed++;
          if (completed === promises.length) {
            resolve(results);     // all done
          }
        })
        .catch(reject);          // any failure rejects all
    });
  });
}
Key insight: Use a counter instead of waiting sequentially. Results are stored by index to preserve original order, not resolution order.
HARD 9. Deep Clone an Object
Write a function to deep clone a JavaScript object without using JSON.parse/stringify — handle arrays, nested objects, and primitive types.
// JSON.parse/stringify fails for:
// - functions, undefined values
// - Date objects (converted to string)
// - circular references
// 
// Build a proper deep clone
function deepClone(value, seen = new WeakMap()) {
  // Handle primitives and null
  if (value === null || typeof value !== 'object') return value;

  // Handle circular references
  if (seen.has(value)) return seen.get(value);

  // Handle Date
  if (value instanceof Date) return new Date(value);

  // Handle Array
  if (Array.isArray(value)) {
    const clone = [];
    seen.set(value, clone);
    value.forEach((item, i) => { clone[i] = deepClone(item, seen); });
    return clone;
  }

  // Handle plain objects
  const clone = {};
  seen.set(value, clone);
  for (const key in value) {
    if (Object.prototype.hasOwnProperty.call(value, key)) {
      clone[key] = deepClone(value[key], seen);
    }
  }
  return clone;
}
Bonus: The WeakMap handles circular references — if we've already started cloning an object, we return the in-progress clone instead of infinite recursing.
